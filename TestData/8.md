  input:        The Markdown file to parse for table of contents,

  -i:           Edit the &lt;input&gt; file directly, injecting the TOC at &lt;!-- toc --&gt;;

  --append:     Append a string to the end of the TOC

<h2><a id="user-content-highlights" class="anchor" aria-hidden="true" href="#highlights"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Highlights</h2>
<p><strong>Features</strong></p>
<ul>
<li>Can optionally be used as a <a href="https://github.com/jonschlinkert/remarkable">remarkable</a> plugin</li>
<li>Returns an object with the rendered TOC (on <code>content</code>), as well as a <code>json</code> property with the raw TOC object, so you can generate your own TOC using templates or however you want</li>
<li>Works with <a href="https://gist.github.com/jonschlinkert/ac5d8122bfaaa394f896">repeated headings</a></li>
<li>Uses sane defaults, so no customization is necessary, but you can if you need to.</li>
<li><a href="#filter-headings">filter</a> out headings you don't want</li>
<li><a href="#titleize">Improve</a> the headings you do want</li>
<li>Use a custom <a href="#optionsslugify">slugify</a> function to change how links are created</li>
</ul>
<p><strong>Safe!</strong></p>
<ul>
<li>Won't mangle markdown in code examples in gfm code blocks that other TOC generators mistake as being actual headings (this happens when markdown headings are show in <em>examples</em>, meaning they arent' actually headings that should be in the toc. Also happens with yaml and coffee-script comments, or any comments that use <code>#</code>)</li>
<li>Won't mangle front-matter, or mistake front-matter properties for headings like other TOC generators</li>
</ul>

<ul>

<h3><a id="user-content-tocinsert" class="anchor" aria-hidden="true" href="#tocinsert"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>toc.insert</h3>
<p>Insert a table of contents immediately after an <em>opening</em> <code>&lt;!-- toc --&gt;</code> code comment, or replace an existing TOC if both an <em>opening</em> comment and a <em>closing</em> comment (<code>&lt;!-- tocstop --&gt;</code>) are found.</p>
<p><em>(This strategy works well since code comments in markdown are hidden when viewed as HTML, like when viewing a README on GitHub README for example).</em></p>
<p><strong>Example</strong></p>
<pre><code>&lt;!-- toc --&gt;
- old toc 1
- old toc 2
- old toc 3
&lt;!-- tocstop --&gt;

## abc
This is a b c.

## xyz
This is x y z.
</code></pre>
<p>Would result in something like:</p>
<pre><code>&lt;!-- toc --&gt;
- [abc](#abc)
- [xyz](#xyz)
&lt;!-- tocstop --&gt;

## abc
This is a b c.

## xyz
This is x y z.
</code></pre>
